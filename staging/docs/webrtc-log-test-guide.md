# WebRTC 연결 로그 대시보드 — 테스트 가이드

사장님 용 — 카메라 끊김·재연결이 실제로 기록되는지 확인하는 절차입니다.

## 1. 사전 준비

- 사장님 계정으로 웹 앱 로그인 (뷰어 측)
- G7 폰 (broadcaster 카메라) 1대 준비 — 페어링 완료 상태
- 두 기기는 같은 홈(home_id)에 속해 있어야 함

## 2. 24시간 방치 테스트

1. G7 폰에서 브라우저로 `/broadcast` 페이지 접속
2. 카메라 코드 입력 → 송출 시작 확인 (PC state: `connected`)
3. **폰을 그대로 두고 24시간 이상 방치** — 화면 꺼짐 OK
4. 중간에 네트워크 불안정 구간이 있으면 더 좋음 (지하철, Wi-Fi 끊김 등)

## 3. 이벤트 이력 확인

1. 사장님 PC/폰에서 웹 앱 로그인
2. `설정 > 📡 WebRTC 연결 로그 보기` 클릭
3. 최근 50건 이벤트가 시간 내림차순으로 표시됨
4. 필터 사용:
   - **역할 필터**: broadcaster / viewer_slot / viewer_live
   - **이벤트 필터**: 에러만 / 재연결만

## 4. 정상 동작 기준

24시간 방치 후 아래 중 **하나 이상** 기록이 있으면 재연결 엔진이 동작한 것:

| 이벤트 | 의미 | 배지 색 |
|--------|------|--------|
| `full_reconnect` (전체 재연결) | 완전 재연결 수행됨 | 파랑 |
| `ice_restart` (ICE 재시작) | 네트워크 경로 재협상 | 파랑 |
| `visibility_reconnect` | 화면 복귀 후 재연결 | 파랑 |
| `connection_recovered` (복구됨) | 끊김 후 자동 복구 성공 | 초록 |

## 5. 문제 진단

- **`failed` / `error` / `keepalive_dead` (빨강)** 이 연속으로 찍히면 재연결 엔진이 실패 중 → 로그 캡쳐해서 개발팀 공유
- **이벤트가 하나도 없음** → broadcaster 로거가 안 붙었거나 RLS 문제 → 확인 필요
- 항목을 **클릭하면 metadata(JSON)** 가 펼쳐짐 — pc_state, reconnect_attempt 등 상세 확인 가능

## 6. 캡처 방법

문제 발견 시:
1. 대시보드에서 해당 이벤트 항목 클릭 → metadata 펼치기
2. 스크린샷 촬영 (시간 + 이벤트 + metadata 다 보이게)
3. Slack `#catvisor-dev` 에 공유
