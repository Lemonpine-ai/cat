## 🅱️-2 다음 세션 #B-2 — YOLO Worker 백엔드 fetch 실패 진단

### 목표 (CRITICAL)

### 사장님 콘솔 로그 (2026-04-25 G7 LTE 실기기 테스트)
```
[yolo-wasm] env ready — wasmPaths= https://cat-lac-eight.vercel.app/ort-wasm/ crossOriginIsolated= true numThreads= 4
Failed to load module script: Expected a JavaScript-or-Wasm module script

[yolo-wasm] wasm backend failed: previous call to 'initWasm()' failed
[YOLO Worker] 모든 백엔드 생성 실패
```

### 핵심 단서 — MIME type 에러
**`.jsep.mjs` 파일이 `application/wasm` 으로 응답** → 브라우저 strict MIME check 거부 → wasm runtime 초기화 실패 → 모든 백엔드 실패.

### 가설 (검증 우선순위)

#### 가설 1 (가장 유력) — Vercel CDN 캐시 옛 MIME 박힘
`next.config.ts` 의 `.mjs` 분리는 정확히 작성됐으나, Vercel CDN edge 가 `Cache-Control: public, max-age=31536000, immutable` (1년) 로 옛 MIME 캐시 보존.

**검증 명령**:
```bash
curl -sI https://cat-lac-eight.vercel.app/ort-wasm/ort-wasm-simd-threaded.jsep.mjs | grep -iE "content-type|cache-control|age"
# 기대: Content-Type: text/javascript; charset=utf-8
# 만약: Content-Type: application/wasm  → 캐시 문제 확정
```

**해결**: Vercel UI 에서 deployment "Purge Cache" 또는 새 path 로 cache busting (helper 파일명에 hash 추가).

#### 가설 2 — path-to-regexp 매치 우선순위
`(.*\.wasm)` 와 `(.*\.mjs)` 둘 다 next.config.ts 에 있음. `.jsep.mjs` 가 어느 정규식에 먼저 매치되는지:
- `.jsep.mjs` 끝이 `.mjs` → 두 번째 룰 매치
- 단 path-to-regexp 가 첫 매치 우선이면 첫 번째 (`.wasm`) 가 잘못 매치할 가능성 (greedy `.*` 동작)

**검증**: 정규식을 `[^/]+\.mjs$` (anchor) 로 강화 → 명확한 매치.

#### 가설 3 — 두 탭 race + Service Worker
어제 사장님이 두 탭 동시 열었을 때도 같은 에러. 같은 URL 동시 fetch race 또는 SW 캐시 충돌. 단일 탭만 열어 재현 시도.

#### 가설 4 — 파일 자체 손상
`public/ort-wasm/ort-wasm-simd-threaded.jsep.mjs` 가 git LFS 또는 빌드 산출물 누락. `git ls-files public/ort-wasm/` 검증.

### 첫 진단 단계 (다음 세션 즉시 실행)

1. **curl 로 production deployment 의 실제 응답 header 확인**:

   ```bash
   curl -sI https://cat-lac-eight.vercel.app/ort-wasm/ort-wasm-simd-threaded.jsep.wasm
   ```
   → Content-Type 비교 + Age header 확인 (CDN 캐시 시간)

2. **`public/ort-wasm/` 파일 존재 여부**:

3. **next.config.ts path-to-regexp 시뮬레이션**:
   - Node.js 로 `path-to-regexp` 라이브러리 import + `(.*\.wasm)` 와 `(.*\.mjs)` 가 `.jsep.mjs` 어떻게 매치하는지 테스트

### 배경
어제 G7 LTE 실기기 테스트에서 60000ms 적용 후에도 일부 세션 timeout 발생:
```
[CameraSlot] 연결 타임아웃 (60000ms) 1ad71734-8304-4706-8bdf-6b8752ca74c6
[CameraSlot] 연결 타임아웃 (60000ms) 55c1133f-555f-4e9e-a053-781faca89dda
```

60초 안에도 ICE 협상이 안 끝남. 가능한 원인:
1. **Cloudflare TURN relay 지연** — 토큰 회전 직후 신규 credential 발급 race
2. **LTE candidate 생성 지연** — 사장님 LTE 폰의 NAT traversal 시간이 60초 이상
3. **가족 4명 동시 시청** — viewer 측 RTCPeerConnection 다중 생성 race
4. **Supabase realtime channel 다중 구독 race** — `[CameraSlot] ICE 채널 구독 성공` 가 여러 번 출력됨

### 우선순위
**Medium** — fix-r3 가 60s 마진 확보로 LTE 성공률 상당 향상. 단 일부 세션 timeout 은 가족 시청 시 사용성 영향.

### 진단 시작 단계
1. ICE candidate type 분포 확인 (`host` / `srflx` / `relay`)
2. Cloudflare TURN credential 발급 timing (Cloudflare 대시보드 → Calls 분석)
3. Supabase channel 구독 race 코드 검토 (CameraSlot.tsx `[CameraSlot] ICE 채널 구독 성공` 중복)
