## 1. 개요

### 1.1 목적

현재 FGS 통증 분석은 Claude Vision API에 의존한다.  
Phase 2A는 **Gemini + Claude 듀얼 채점**으로 학습 데이터를 대량 확보하여  
**MobileNetV3-Small 온디바이스 모델을 실전 수준으로 학습**시키는 것이 목표다.

### 1.2 핵심 아이디어: 큰 모델이 선생, 작은 모델이 학생

```
[큰 모델 = 선생님]          [작은 모델 = 학생]
Gemini Flash + Claude   →   MobileNetV3-Small
  (채점 능력 뛰어남)          (휴대폰에서 돌아감)
  (비용 발생)                 (비용 $0)
```

- 선생 모델(Gemini + Claude)이 수천장의 고양이 사진에 FGS 점수를 자동 매김
- 학생 모델(MobileNetV3)이 그 점수로 학습 → 선생의 판단력을 물려받음
- 교차 검증 + 사장님 눈 확인으로 라벨 품질 보장

### 1.3 왜 Gemini인가?

| 항목 | Gemini Flash | Claude Sonnet |
|------|-------------|---------------|
| 비용 | **무료 티어 1,500건/일** | 이미지당 $0.003~0.005 |
| Vision 성능 | 좋음 (고양이 표정 인식 가능) | 좋음 (현재 프로덕션 사용 중) |
| 역할 | **1차 채점 (무료 대량 처리)** | **2차 교차 검증 (선별 채점)** |

→ Gemini를 1차로 돌려 비용을 대폭 줄인다.

### 1.4 범위 한정

- **Phase 2A 범위**: Colab에서 데이터 수집 + 채점 + 학습까지만
- **Cat ID(개체 식별)**: Phase 2B 범위. 이 설계서에서는 **통증 분류(Head B)만** 다룸
- **앱 탑재**: Phase 2A 성공 후 별도 설계서 작성

```

└── 수의 공개 자료 → 추가 통증 데이터 수집
        │
        ▼  [게이트 1: 정상 3,000장 + 통증 200장 이상?]
        │

├── Gemini Flash → 전체 이미지 1차 채점 (무료)
│     │
│     ▼  [게이트 2: 첫 100장 환각률 ≤ 20%?]
│     │
├── Claude Sonnet → 선별 이미지 2차 채점 (~$5)
└── 결과: results.csv [파일명, gemini_score, claude_score, 합의여부]
        │
        ▼
Phase C: 필터링 + 사람 검수 (사장님 일정에 따라)
 → 한 줄 요약: 정상은 자동 채택, 통증 의심분만 사장님이 눈으로 확인
├── 정상 (둘 다 0~1): 자동 채택
├── 통증 의심 (둘 중 하나 2+): 사장님 수동 검수
└── 불일치 (차이 ≥ 2): 폐기
        │
        ▼  [게이트 3: 검수 완료, 통증 데이터 300장+?]
        │

├── 데이터 분할: 학습 80% / 검증 10% / 테스트 10%
│   ※ 기존 통증 30장 = 고정 테스트셋 (절대 학습에 안 씀)
├── 통증 데이터가 적으니 통증 사진을 더 자주 보여주는 방식 적용
├── 학습: 50 에포크, 성능 안 오르면 자동 중단
└── 저장: Google Drive /catvisor-ai/models/mobilenetv3-fgs/
        │
        ▼  [게이트 4: 통증 Recall ≥ 80%, 정상 Precision ≥ 90%?]
        │
    ✅ 성공 → ONNX + TFLite 변환 → CATvisor 앱 탑재 (별도 설계)
    ❌ 실패 → Claude Vision API 유지 (현재 프로덕션 그대로)
```

## 4. Phase A: 데이터 수집

### 4.1 데이터 소스

| 소스 | 예상 수량 | 용도 | 라이선스 |
|------|----------|------|---------|
| Roboflow Cat Face 데이터셋 | ~8,000장 | 정상 고양이 (이미 얼굴 크롭됨) | **수집 시 확인 필수** |
| Kaggle Cat Emotions | ~1,400장 | 정상 + 약간의 불편 표정 | **수집 시 확인 필수** |
| 기존 학습 데이터 | 1,360 + 90장 | 정상 + 통증 | 자체 데이터 |
| Augmentation (통증 60장 기반) | 60 → 300장 | 통증 데이터 5배 확장 | - |

> **주의**: 구체적인 데이터셋 ID/URL은 Phase A 노트북 제작 시 확정. Roboflow는 `rf.workspace("워크스페이스명").project("프로젝트명")` 형식 필요 (이전 학습에서 확인된 사항).

### 4.2 데이터 품질 기준 (수집 시 필터링)

- [ ] 고양이 얼굴이 정면~45도 이내로 보일 것
- [ ] 해상도 224px x 224px 이상
- [ ] 사진 당 고양이 1마리만
- [ ] 심하게 흐리거나 역광인 사진 제외
- [ ] 사람 손/물체가 얼굴을 가리는 사진 제외

### 4.3 라이선스 체크리스트

수집하는 모든 데이터셋에 대해 확인:

- [ ] 상업적 사용 가능한 라이선스인가? (CC-BY, MIT, Apache 등)
- [ ] CC-BY-SA라면 → 모델 배포 시 출처 표기 필요 (기록해둘 것)
- [ ] CC-BY-NC(비상업)라면 → 사용 불가, 대체 데이터셋 탐색

### 4.4 EXIF 메타데이터 제거

수집한 모든 이미지는 **EXIF 메타데이터를 제거**한 후 저장한다.

- 이유: 외부 수집 이미지에 촬영자 정보, 위치 정보 등이 포함될 수 있음
- 방법: Pillow `image.save()` 시 exif 파라미터 제거 (노트북에서 자동 처리)

### 4.5 테스트셋 사전 분리

**Phase A 시작 시** 기존 통증 90장을 즉시 분리:

- **60장** → 학습용 (augmentation 대상)
- **30장** → 테스트 전용 (이후 절대 학습에 사용 안 함, `data/test_holdout/` 별도 보관)

### 4.6 Augmentation 전략 (통증 60장 → 300장)

| 변환 | 파라미터 | 이유 |
|------|---------|------|
| 수평 반전 | 50% 확률 | 좌우 대칭은 FGS에 영향 없음 |
| 밝기 조절 | ±20% | 다양한 조명 환경 대응 |
| 회전 | ±15도 | 약간의 카메라 각도 차이 대응 |
| 크롭 + 리사이즈 | 80~100% 영역 | 얼굴 위치 다양화 |

**금지 변환**: 색상 반전, 과도한 왜곡, 모자이크 (FGS 판단에 필요한 표정 특성 훼손)

## 5. Phase B: 듀얼 채점

### 5.1 프롬프트 전략: 수의사 전문가 페르소나 + FGS 정량 기준 + 종합 소견

**3단 구조**로 구성:

1. **역할 부여**: "40년 경력의 고양이 표정 전문 수의사. 귀·눈·입·수염 모양만 보고 통증을 즉시 알아차림" 페르소나
2. **정량 기준**: `fgsPrompt.ts`의 FGS 5-AU 채점 기준 전문 포함 (카메라 앵글 보정, 흔한 실수 등 포함)
3. **종합 소견**: AU 채점 외 `clinical_note` 필드에 임상 경험 기반 종합 판단 추가

**핵심 원칙**: Gemini와 Claude에게 **동일한 프롬프트**를 사용한다.

응답 JSON 형식:

```json
{
  "fgs_score": 0,
  "confidence": 0.85,
  "au_scores": { "ear": 0, "eye": 0, "muzzle": 0, "whisker": 0, "head": 0 },
  "reasoning": "각 AU별 판단 근거 (한국어)",
  "clinical_note": "임상 경험 기반 전반적 상태 종합 소견 (한국어)"
}
```

- `clinical_note`는 사장님 수동 검수(Phase C) 시 참고용
- 프롬프트가 다르면 교차 검증 의미 없음. **반드시 동일 프롬프트 사용.**

### 5.2 트리플 채점 — 3심 다수결 (완전 자동화)

사장님 수동 검수 없이 **3개 AI 심판의 다수결**로 자동 라벨 확정:

| 심판 | 모델 | 비용 | 역할 |
|------|------|------|------|
| 1심 | Gemini Flash | 무료 | 전수 1차 채점 |
| 2심 | Claude Sonnet | ~$5 | 통증 의심분 + 10% 확인 |
| 3심 | GPT-4o | ~$1.5 | 통증 의심분 + 10% 확인 |

```
전체 이미지
    │
    ▼
[1단계] Gemini Flash 전수 채점 (무료)
    │
    ├── Gemini 0~1점 → 10%만 Claude + GPT-4o 확인
    │
    ├── Gemini 2~4점 → Claude + GPT-4o 전수 채점 → 3심 다수결
    │   2/3 정상 → auto_normal
    │   2/3 통증 → auto_pain
    │   전부 다름 → 폐기
    │
    ├── Gemini 에러 → Claude + GPT-4o 2심 판정
    │
    └── confidence 평균 < 0.6 → 폐기
```

**핵심**: 3개 모델이 **동일 프롬프트, 독립 판단**. 서로 결과를 모름.

### 5.3 Gemini API 호출 제약

| 항목 | 값 |
|------|-----|
| 모델 | `gemini-2.5-flash` |
| 무료 티어 | 15 RPM, 1,500 RPD |
| 5,000장 처리 시 | 약 3~4일 분산 |
| 중간 저장 | Drive에 `results_partial.csv` 매 100건마다 저장 |
| 재시작 | 이미 채점된 파일명은 건너뜀 (같은 작업 다시 돌려도 중복 안 됨) |

### 5.4 환각 체크 (게이트 2)

Phase B 시작 후 **처음 100장은 사장님이 수동 확인**:

- Gemini가 정상 고양이를 통증 2+ 로 채점한 비율 측정
- **환각률 > 20%** → Gemini 단독 자동 채택 중단, 전수 Claude 교차로 전환
- **환각률 ≤ 20%** → 계획대로 진행

## 6. Phase C: 자동 필터링 (3심 다수결 기반, 사람 검수 불필요)

### 6.1 3심 다수결 자동 필터링

| 조건 | 처리 | 이유 |
|------|------|------|
| 3심 중 2/3 이상 정상(0~1) | **자동 정상** | 다수결 합의 |
| 3심 중 2/3 이상 통증(2+) | **자동 통증** | 다수결 합의 |
| Gemini 단독 정상 (10% 확인 안 걸림) | **자동 정상** | G2 환각 체크 통과 전제 |
| 3모델 전부 다른 점수 | **폐기** | 불일치 → 노이즈 |
| confidence 평균 < 0.6 | **폐기** | 확신 부족 |
| 2모델 미만 채점 | **폐기** | 판단 불가 |

### 6.2 사장님 수동 검수: 불필요

3심 다수결로 완전 자동화. **사장님 시간 0분.**
안전장치는 Phase D의 **게이트 4(테스트셋 30장)**가 담당.

## 7. Phase D: MobileNetV3 학습

### 7.1 모델 구조

```
MobileNetV3-Small (ImageNet 사전학습)
    │
    ├── backbone (특징 추출) → 처음 30 에포크 동안 고정
    │
    └── classifier head (새로 교체)
        ├── Linear(576, 128) + ReLU + Dropout(0.3)
        └── Linear(128, 2)   → [정상, 통증]
```

> Cat ID용 Head A(임베딩 128D)는 **Phase 2B 범위**. Phase 2A에서는 통증 분류 Head B만 학습한다.

### 7.2 2클래스 매핑

기존 5클래스(FGS 0~4) → 2클래스로 단순화:

| FGS 점수 | 클래스 | 이유 |
|----------|--------|------|
| 0, 1 | **정상 (0)** | 관찰 불필요 |
| 2, 3, 4 | **통증 (1)** | 수의사 상담 권장 |

→ MobileNetV3는 **"통증 있다/없다"만 판단** (빠른 선별용).  
→ 세부 점수(0~4)는 통증 감지 시 Claude API가 담당.  
→ 데이터가 500건+ 쌓이면 향후 5클래스로 확장 가능.

### 7.3 데이터 분할

```
[학습 데이터]
├── 듀얼 채점 + 필터링된 정상 데이터: ~3,000장
├── 듀얼 채점 + 사장님 검수 통증 데이터: ~300장
├── 기존 통증 60장 (90장 중 학습용)
└── Augmentation 통증 데이터: ~240장 (60장 × 4변환)

[검증 데이터] ── 학습 데이터에서 10% 분리

[테스트 데이터 (절대 학습에 사용 안 함)]
├── 기존 통증 30장 (Phase A에서 분리한 고정 테스트셋)
└── 정상 150장 (무작위 추출, 고정)
```

### 7.4 학습 설정 (개발자 참고)

> 이 섹션은 2번 Dev 구현 참고용입니다. 사장님은 넘어가셔도 됩니다.

| 항목 | 값 | 한줄 설명 |
|------|-----|----------|
| Optimizer | AdamW | 학습 최적화 방식 |
| Learning rate | 1e-4 → 3e-4 | 처음엔 천천히, 나중엔 빠르게 |
| Batch size | 32 | 한 번에 32장씩 학습 |
| Epochs | 50 (patience=10) | 최대 50회 반복, 성능 안 오르면 자동 중단 |
| WeightedRandomSampler | 정상:통증 = 1:3 | 통증 사진을 3배 자주 보여줌 (데이터 불균형 보정) |
| class_weights | [1.0, 5.0] | 통증 놓치면 5배 페널티 (의료 특성상 놓침이 더 위험) |
| 이미지 크기 | 224 x 224 | MobileNetV3 표준 입력 |
| 랜덤 시드 | 42 | 다시 돌려도 같은 결과 나오게 (재현성) |

### 7.5 학습 스케줄 (개발자 참고)

```
전반부 (에포크 1~30): backbone 고정
  - 새로 붙인 head만 학습 → 빠르게 기본 판단력 형성

후반부 (에포크 31~50): 전체 학습
  - 미세한 고양이 표정 특징까지 학습
  - 학습률 점점 줄여가며 세밀하게 조정
```

### 7.6 모델 변환

학습 완료 후 두 가지 형식으로 변환:

| 형식 | 용도 | 비고 |
|------|------|------|
| **ONNX** | 브라우저 / 서버 | ONNX Runtime Web으로 브라우저에서 실행 |
| **TFLite** | 모바일 앱 | 향후 네이티브 앱 대비 |

### 7.7 GPU 미할당 시 대응

Colab 무료 티어에서 T4 GPU를 못 받는 경우:
- 런타임 재시작 시도 (보통 몇 분 내 할당)
- 반복 실패 시 → 다음 날 재시도 (새벽 시간대 할당 확률 높음)
- CPU로도 학습 가능하지만 시간이 10배 이상 걸림 → 비추천

## 8. Go/No-Go 게이트 + 리스크

### 8.1 게이트

| 게이트 | 위치 | 통과 조건 | 실패 시 대응 |
|--------|------|----------|-------------|
| **G1** | Phase A → B | 정상 ≥ 3,000장, 통증 ≥ 200장 | 추가 데이터 소스 탐색 |
| **G2** | Phase B 100장 후 | Gemini 환각률 ≤ 20% | 전수 Claude 교차 검증 전환 (비용 증가) |
| **G3** | Phase C → D | 사장님 검수 완료, 통증 ≥ 300장 | augmentation 추가 또는 데이터 수집 연장 |
| **G4** | Phase D 후 | Recall ≥ 80%, Precision ≥ 90% | **배포 안 함, Claude API 유지** |

**G4 실패는 프로젝트 실패가 아니다.** 현재 Claude Vision API가 잘 돌아가고 있으므로 그대로 유지하면 된다. 데이터 더 모은 후 재시도.

### 8.2 리스크

| 리스크 | 확률 | 대응 |
|--------|------|------|
| Gemini 무료 티어 축소 | 낮음 | Phase B를 가능한 빨리 실행 |
| Gemini safety filter가 아픈 고양이 사진 차단 | 중간 | 차단된 이미지는 Claude 단독 채점 fallback |
| 통증 데이터 300장 미달 | 중간 | augmentation 강도 높임 + 수의 자료 추가 수집 |
| 두 모델 모두 같은 방향 오판 | 중간 | 사장님 수동 검수 + 테스트셋 30장으로 최종 확인 |
| 사장님 검수 "모르겠음" 비율 높음 | 중간 | "모르겠음"은 폐기, G3에서 수량 체크 |
| Colab 런타임 끊김 | 높음 | 100건마다 중간 저장, 같은 작업 다시 돌려도 중복 안 됨 |
| G4 실패 (성능 미달) | 중간 | **실패해도 OK.** Claude API 유지 |

## 9. 성공 기준

### 9.1 모델 성능 기준 (테스트셋 기준)

| 지표 | 목표 | 의미 |
|------|------|------|
| **통증 Recall** | **≥ 80%** | 통증 고양이 10마리 중 8마리 이상 잡아냄 |
| **정상 Precision** | **≥ 90%** | "정상"이라고 한 것 중 90% 이상이 진짜 정상 |
| **전체 Accuracy** | ≥ 85% | 전체적으로 85% 이상 맞춤 |
| **F1-score (통증)** | ≥ 0.75 | 통증 판별 종합 성능 |

### 9.2 의료 안전 장치

MobileNetV3가 배포되더라도:

- 온디바이스 모델 = **스크리닝(1차 선별)** 용도
- 통증 감지 시 → Claude Vision API로 **2차 확인** (확진용)
- 사용자에게는 **"AI 참고 수치입니다. 정확한 진단은 수의사에게"** 안내 표시
- 즉, 온디바이스 모델이 "통증 있다" → Claude가 재확인 → 사용자에게 알림
